# Domain Document: Windows File Explorer

## 1. 목적

이 문서는 Windows용 파일 탐색기 애플리케이션의 핵심 기능, 도메인 용어, 유스케이스 경계, Windows Shell 연동 규칙을 정의한다.

애플리케이션은 사용자가 로컬 파일 시스템, Windows 기본 폴더, 드라이브, 네트워크 공유 경로를 탐색하고 파일 작업을 수행할 수 있도록 한다. 파일 복사, 이동, 삭제, 이름 변경, 실행, 속성 보기처럼 Windows와 강하게 결합된 작업은 자체 구현보다 Windows Shell 시스템 기능을 우선 사용한다.

## 2. 범위

### 포함 범위

- 폴더 탐색과 파일/폴더 목록 표시
- 새 폴더 생성
- 파일/폴더 복사, 이동, 삭제, 이름 변경
- 파일 실행과 파일 속성 보기
- 목록 정렬과 자세히 보기 모드
- 다중 탭 탐색
- 주소 표시줄과 탐색 버튼
- 왼쪽 폴더 트리
- 북마크와 빠른 접근
- 북마크 추가, 이름 변경, 삭제, 순서 변경
- 드라이브와 Windows 기본 폴더 접근
- 네트워크 공유 경로 접근
- Windows 연결 프로그램, 컨텍스트 메뉴, 휴지통, 파일 아이콘 연동
- 숨김 파일과 시스템 파일 표시 옵션
- 외형 테마와 글꼴 선택 및 저장
- Windows DPI awareness와 DPI 기준 UI 크기 보정
- 실행 인자로 전달된 시작 경로 열기
- 저장된 시작 폴더 옵션과 기본 시작 후보
- 현재 폴더 검색과 하위 폴더 포함 검색
- 파일명 기반 검색
- 검색 취소
- 파일/폴더 드래그앤드롭 복사와 이동
- 외부 프로그램의 현재 폴더 파일/폴더 변경 감지와 목록 갱신
- 파일 작업 실패 시 내부 오류와 사용자 메시지 분리
- 애플리케이션 창과 실행 파일 아이콘 표시
- 앱 정보 창에서 버전 정보, 프로젝트 링크, 라이선스 고지 표시

### 제외 범위

- 클라우드 동기화 엔진 직접 구현
- 파일 시스템 드라이버 또는 가상 파일 시스템 직접 구현
- 압축 파일 내부 탐색
- 파일 내용 인덱싱 기반 전문 검색
- Windows Shell 동작을 대체하는 자체 충돌 해결 UI

## 5. 핵심 유스케이스

### 5.0 실행 경로 인자 처리

사용자는 애플리케이션 실행 시 Windows 탐색기처럼 시작 경로를 전달할 수 있다.

규칙:

- 시작 경로 인자가 없을 때 앱은 Windows 폴더 선택 대화상자를 자동으로 띄우지 않는다.
- 사용자는 탭 메뉴에서 현재 폴더를 시작 폴더로 저장하거나 저장된 시작 폴더를 해제할 수 있다.
- 시작 폴더를 설정하면 이전 탭 복원 옵션은 꺼지고, 이전 탭 복원 옵션을 켜면 저장된 시작 폴더는 해제된다.
- 시작 경로 인자가 있으면 저장된 시작 폴더와 이전 탭 복원보다 해당 인자를 우선한다.
- 시작 경로 인자가 없고 시작 폴더 설정이 있으면 해당 폴더를 시작 후보의 첫 번째 위치로 사용한다.
- 저장된 시작 폴더에 접근할 수 없으면 기본 시작 후보를 사용한다.
- 시작 경로 인자와 시작 폴더 설정이 모두 없으면 기본 시작 후보를 사용한다.
- 폴더 경로 인자는 해당 폴더를 현재 탭의 시작 위치로 연다.
- 파일 경로 인자는 부모 폴더를 시작 위치로 열고 파일을 시작 선택 항목으로 둔다.
- `/select,<path>` 형식은 대상이 파일이든 폴더든 부모 폴더를 열고 대상을 시작 선택 항목으로 둔다.
- 시작 경로 인자가 존재하지 않거나 부모 폴더를 열 수 없으면 사용자에게 시작 위치 오류를 표시한다.

### 5.0.1 앱 정보 표시

사용자는 About 메뉴에서 앱 정보를 확인할 수 있다.

규칙:

- 앱 정보 창은 현재 실행 중인 패키지 버전을 표시한다.
- 앱 정보 창 제목은 `About j3Files`이고, 상단에는 `j3Files {version}` 형식의 버전 라벨을 표시한다.
- 앱 정보 창은 실행 파일과 같은 디렉터리의 `about.txt` 내용을 우선 표시한다.
- 배포 파일의 `about.txt`를 읽을 수 없거나 비어 있으면 빌드에 포함된 기본 `about.txt` 내용을 표시한다.
- `about.txt` 본문은 읽기 전용 멀티라인 영역에 표시하고, 내용이 길어져도 스크롤 가능해야 한다.
- 앱 정보 창은 프로젝트 저작권 표시, `GPL-3.0-or-later` 프로젝트 라이선스, `LICENSE` 전문 위치, `THIRD_PARTY_NOTICES.txt` 고지 위치, 바이너리 배포 시 대응 소스코드 제공 안내를 포함한다.
- 앱 정보 창은 본문 길이에 맞춰 과도하게 커지지 않도록 고정 폭의 compact 대화상자로 표시한다.
- 앱 정보 창 하단에는 `https://github.com/edgarp9` 프로젝트 링크를 표시한다.
- 링크 열기는 Windows Shell 기본 브라우저 동작을 사용하며, 앱의 탐색 상태를 변경하지 않는다.

### 5.1 폴더 탐색

사용자는 현재 탭에서 폴더, 드라이브, Windows 기본 폴더, 네트워크 경로로 이동할 수 있다.

기본 흐름:

1. 사용자가 주소 표시줄, 목록 항목, 빠른 접근, 북마크, 드라이브 목록, 탐색 버튼 중 하나로 이동을 요청한다.
2. 애플리케이션은 요청된 위치를 탐색 위치로 해석한다.
3. 접근 가능하면 현재 탭의 현재 경로를 갱신한다.
4. 현재 폴더의 파일/폴더 목록을 자세히 보기 모드로 표시한다.
5. 이동 방식에 따라 탭의 뒤로/앞으로 기록을 갱신한다.

예외 흐름:

- 경로가 존재하지 않으면 사용자에게 위치를 찾을 수 없다는 메시지를 표시한다.
- 접근 권한이 없으면 권한 문제를 설명하고 가능한 경우 Windows 권한 상승 흐름을 사용한다.
- 네트워크 경로에 연결할 수 없으면 네트워크 위치 접근 실패 메시지를 표시한다.

### 5.2 파일/폴더 목록 표시

사용자는 현재 폴더의 파일과 폴더를 목록으로 확인할 수 있다.

규칙:

- 기본 보기 모드는 자세히 보기이다.
- 목록에는 이름, 크기, 갱신일, 종류 정보를 표시한다.
- 파일 항목은 내부 경로, 표시 이름, 파일/폴더 항목 구분, 종류 이름, 크기, 갱신일, 숨김/시스템/읽기 전용 속성을 가진다.
- 초기 구현의 종류 이름은 폴더/드라이브/네트워크 공유의 고정 라벨과 파일 확장자 기반 라벨을 사용한다.
- 폴더 항목은 파일 항목보다 먼저 표시하고, 같은 그룹 안에서는 정렬 기준에 따라 안정적으로 정렬한다.
- 같은 정렬 키 값을 가진 항목은 디렉터리 조회 결과의 기존 순서를 보존한다.
- 숨김 파일 표시 옵션이 꺼져 있으면 숨김 파일은 표시하지 않는다.
- 시스템 파일 표시 옵션이 꺼져 있으면 시스템 파일은 표시하지 않는다.
- 파일 아이콘은 Windows Shell에서 제공하는 아이콘을 사용한다.
- 목록에 남은 파일 항목이 Windows Explorer 등 외부 프로그램에서 이미 삭제된 상태로 열기, 연결 프로그램 선택, 속성, 이름 변경, 삭제, 컨텍스트 메뉴, 폴더 진입을 요청하면 앱은 먼저 사용자에게 경고를 표시한다. 오류 대상이 현재 목록 항목과 일치하고 실제 경로가 더 이상 존재하지 않음이 확인되면 경고 후 해당 항목을 현재 목록과 선택 상태에서 제거한다.
- 앱은 현재 활성 탭의 실제 탐색 위치를 백그라운드에서 감시한다. Windows Explorer 등 외부 프로그램이 해당 폴더의 파일 또는 하위 폴더를 생성, 삭제, 이름 변경, 속성 변경, 크기 변경, 갱신일 변경하면 변경 알림을 받은 뒤 짧게 debounce하고 현재 목록을 다시 조회한다.
- 외부 변경 감시는 이벤트 내용을 직접 증분 반영하지 않고, 영향을 받은 폴더가 변경되었다는 신호로만 사용한다. 이는 rename old/new 이벤트 분리, 대량 변경, 버퍼 overflow 가능성 때문에 목록의 최종 상태를 파일 시스템 재조회로 맞추는 편이 안정적이기 때문이다.
- 활성 탭 또는 현재 위치가 바뀌면 이전 외부 변경 감시 worker는 취소하고 새 위치를 감시한다. 이전 worker에서 늦게 도착한 변경 알림은 generation으로 구분해 무시한다.

### 5.3 새 폴더 생성

사용자는 현재 폴더 안에 새 폴더를 생성할 수 있다.

규칙:

- 새 폴더 생성은 현재 탐색 위치가 쓰기 가능한 파일 시스템 위치일 때만 허용한다.
- 새 폴더 이름은 단일 폴더 이름이어야 하며 경로 구분자, Windows에서 허용하지 않는 문자, 제어 문자, 예약 장치 이름, 끝의 공백 또는 마침표를 허용하지 않는다.
- 새 폴더 이름 검증은 `domain`의 순수 규칙으로 먼저 수행하고, 실제 생성 가능 여부는 `infra/platform`의 파일 시스템 생성 결과로 판정한다.
- 이름 충돌, 권한 부족, 잘못된 이름은 Windows Shell 또는 파일 시스템 오류에 근거해 사용자 메시지로 변환한다.
- 새 폴더 생성 후 목록을 갱신하고 생성된 폴더를 선택 상태로 둘 수 있다.

### 5.4 복사, 이동, 삭제, 이름 변경

사용자는 파일 또는 폴더를 복사, 이동, 삭제, 이름 변경할 수 있다.

규칙:

- 기본 구현은 Windows Shell의 `IFileOperation`을 사용한다.
- 충돌 처리, 권한 상승, 진행률, 취소는 Windows Shell 동작을 우선 사용한다.
- 삭제는 기본적으로 휴지통 이동으로 처리한다.
- 즉시 삭제가 별도 기능으로 추가되는 경우, 휴지통 삭제와 명확히 구분해야 한다.
- `app` 계층은 선택 항목과 대상 경로를 받아 Shell 파일 작업 유스케이스를 조합하고, 작업 완료 후 갱신해야 할 영향을 받은 폴더 목록을 반환한다.
- 실패 시 내부 오류에는 Windows 오류 코드와 작업 종류를 보존하고, 사용자에게는 이해 가능한 메시지를 표시한다.

### 5.4.1 드래그앤드롭 복사와 이동

사용자는 파일 목록의 파일 또는 폴더를 다른 폴더로 끌어 놓거나, Windows Shell에서 끌어 온 파일 또는 폴더를 현재 탐색 위치로 놓을 수 있다. 일반 폴더 목록에서 시작한 선택 항목은 Windows Explorer, Desktop, `CF_HDROP`를 이해하는 Shell 호환 앱으로 끌어낼 수 있다.

기능:

- 파일 목록 선택 항목을 같은 앱의 파일 목록 폴더 항목으로 드롭
- 파일 목록 선택 항목 또는 왼쪽 폴더 트리 항목을 왼쪽 폴더 트리 항목으로 드롭
- 왼쪽 폴더 트리 항목을 파일 목록의 폴더 항목 또는 Windows Explorer, Desktop, `CF_HDROP` 기반 Shell 호환 앱으로 끌어내기
- Windows Explorer 등 외부 Shell 파일 소스를 현재 파일 목록 빈 영역 또는 폴더 트리 항목으로 드롭
- 파일 목록 선택 항목을 Windows Explorer, Desktop, `CF_HDROP` 기반 Shell 호환 앱으로 끌어내기
- `Ctrl` 드롭은 복사, `Shift` 드롭은 이동으로 처리
- 수정 키가 없을 때 앱 내부 드롭은 Windows Explorer 기대와 맞춰 같은 드라이브 또는 같은 UNC 공유 안에서는 이동, 다른 드라이브 또는 다른 UNC 공유로는 복사를 기본값으로 사용

규칙:

- 드래그 소스는 하나 이상의 `NavigationLocation`이어야 하며 표시 문자열에서 경로를 역변환하지 않는다.
- 드롭 대상은 폴더, 드라이브, Windows 기본 폴더, 네트워크 공유처럼 탐색 가능한 폴더 위치여야 한다.
- 파일 목록 빈 영역에 드롭하면 현재 활성 탭의 현재 위치를 대상 폴더로 사용한다.
- 파일 목록의 폴더 항목 위에 드롭하면 해당 폴더 항목을 대상 폴더로 사용한다.
- 왼쪽 폴더 트리 항목 위에 드롭하면 해당 트리 항목의 `NavigationLocation`을 대상 폴더로 사용한다.
- 파일 항목, 검색 결과 행, 검색 상태 행, 탭바, 주소 표시줄, 버튼 영역에 대한 파일 드롭은 현재 구현에서 허용하지 않는다.
- 검색 결과 표시 중 내부 드래그 소스로 파일 작업을 시작하는 흐름은 현재 구현에서 제외한다. 검색 결과 표시 중 외부 Shell 드롭은 파일 목록 빈 영역에서만 허용하며 현재 탭의 실제 현재 위치를 대상 폴더로 사용한다.
- 대상 폴더 접근 가능성은 `app` 계층에서 확인하고, 실제 쓰기 가능 여부와 충돌 처리는 Windows Shell 파일 작업 결과에 따른다.
- 이동 드롭에서 대상이 소스 자신이거나 소스의 하위 경로이면 Shell 호출 전에 거부한다.
- 이미 다른 파일 작업 worker가 실행 중이면 새 드롭 파일 작업을 시작하지 않고 기존 파일 작업 진행 중 메시지를 표시한다. 이 동안 OLE 드롭 대상 피드백은 불가 효과로 갱신하고, 내부 파일 목록 드래그 시작도 거부한다.
- 드롭으로 실행되는 복사/이동은 붙여넣기와 같은 `IFileOperation` 경로를 사용하고 Shell 진행률, 충돌 해결, 권한 상승, 취소 동작을 재사용한다.
- 완료 후 복사는 대상 폴더를, 이동은 소스 부모 폴더와 대상 폴더를 갱신 대상으로 반환한다.
- 현재 활성 탭의 위치가 갱신 대상에 포함되면 목록을 새로 고치고, 폴더 구조가 바뀔 수 있으므로 폴더 트리도 갱신한다.
- 작업이 취소되면 오류 대화상자를 표시하지 않는다. 실패하면 내부 오류와 사용자 메시지를 분리해 표시한다.
- 드롭으로 생성된 복사/이동 작업도 안전하게 추적 가능한 경우 `Ctrl+Z` 실행 취소 후보가 될 수 있다.
- 앱 밖으로 끌어내는 Shell drag source는 파일 목록의 현재 선택 항목 또는 왼쪽 폴더 트리 항목을 `NavigationLocation` 목록으로 스냅샷한 뒤, `IDataObject`에 내부 드래그 식별자와 `CF_HDROP` 파일 목록을 제공한다.
- 드래그 소스 목록은 Windows 경로 비교 기준으로 중복을 제거하고 첫 번째 항목 순서를 보존한다. 표기 대소문자만 다른 같은 경로는 같은 소스로 취급한다.
- 앱 밖 Shell drag source는 `DoDragDrop`을 시작하기 전에 `CF_HDROP` 항목 수 4,096개 이하, 항목별 UTF-16 경로 길이 32,767 코드 유닛 이하, 전체 `HGLOBAL` 크기 overflow 가능성, 전체 `HGLOBAL` 64 MiB 이하 정책을 검증하고, 실패하면 사용자 메시지로 복구 가능한 오류를 반환한다.
- 앱 밖 Shell drag source는 목적지를 알 수 없으므로 `Preferred DropEffect`를 제공하지 않고, 허용 효과 copy/move와 `CF_HDROP`만 제공해 Explorer/Desktop 같은 드롭 대상 Shell이 같은 드라이브 이동, 다른 드라이브 복사, UNC 공유 정책을 결정하게 한다.
- 앱 밖으로 끌어낸 파일의 실제 복사/이동은 드롭 대상 Shell이 수행하며, 앱은 `DoDragDrop` 결과를 취소, 드롭 없음, 복사 완료, 이동 완료 가능성으로 구분한다.
- 앱 밖 Shell 드래그에서 이동 완료 가능성이 있으면 원본 부모 폴더를 갱신 대상으로 계산하고, 현재 탭 위치가 포함되면 목록과 폴더 트리를 갱신한다.
- 외부 Shell 드롭 수신 시 `Preferred DropEffect`가 있으면 허용 효과와 함께 기본 copy/move 판단에 반영한다. 선호 효과가 없고 Shell 허용 효과가 copy/move 둘 다이면 앱의 안전한 기본값으로 복사를 사용한다.
- 외부 Shell 드롭 수신은 `CF_HDROP` 파일 시스템 경로를 대상으로 한다. Shell namespace/PIDL만 제공하고 파일 시스템 경로를 제공하지 않는 고급 드래그 소스는 현재 구현에서 지원하지 않는다.
- 외부 Shell 드롭 수신도 `CF_HDROP` 항목 수 4,096개 초과, 항목별 경로 길이 32,767 UTF-16 코드 유닛 초과, 전체 파일 목록 64 MiB 초과를 거부한다.
- 존재하지 않는 경로, 접근 불가 경로, 파일 사용 중 상태는 대량 선택 사전 검증에서 모든 항목을 동기 stat하지 않고 `IFileOperation` 또는 Shell 드롭 대상 결과로 판정한다. 이는 네트워크/UNC 대량 선택에서 UI thread가 오래 멈추는 것을 피하기 위한 정책이다.
- UNC 경로는 같은 `\\server\share` 또는 verbatim UNC 공유 안에서 앱 내부 기본 이동, 서로 다른 UNC 공유에서는 앱 내부 기본 복사를 사용한다. 앱 밖으로 끌어낸 경우 기본 효과는 대상 Shell 정책에 맡긴다.
- 권한 부족, UAC 권한 상승, 이름 충돌, 긴 경로 정책, 파일 사용 중 오류는 앱이 직접 대체 UI를 만들지 않고 `IFileOperation` 결과와 Shell 오류 매핑에 따른 사용자 메시지로 처리한다.
- OLE drop 이벤트를 앱 창 메시지 큐로 전달할 수 없으면 성공 효과를 반환하지 않고 드롭 없음으로 복구한다.
- 파일 작업 worker 완료 후 UI 갱신은 copy 대상 폴더, move 원본 부모와 대상 폴더, delete 원본 부모처럼 실제 작업 영향 폴더를 기준으로 판단한다.
- 파일 작업 worker 완료 후 왼쪽 폴더 트리는 전체 루트를 삭제하고 다시 만들지 않고, 실제 영향 폴더와 일치하는 이미 표시된 TreeView 항목의 자식 폴더만 다시 조회한다.
- 파일 작업 시작과 완료 후 재조회 중에는 오른쪽 파일 목록을 진행/로딩 상태 행으로 대체하지 않고 기존 목록을 유지한다. 진행 중/종료 대기 메시지는 별도 파일 작업 상태 줄에 표시하고, 새 목록 조회 결과가 도착하면 ListView 행을 한 번에 교체한다.
- 파일 작업 완료나 외부 변경 감지처럼 같은 탐색 위치와 표시 옵션을 다시 조회해 오른쪽 파일 목록 행을 교체하는 경우에는 탐색 이동으로 간주하지 않고 기존 ListView 뷰포트의 top index를 보존한다.
- ListView/TreeView 행을 일괄 교체하기 위해 redraw를 일시 중지했다가 재개할 때는 컨트롤 전체 배경 erase를 강제하지 않는다. 두 컨트롤은 자체 배경을 페인트하므로 erase를 먼저 요청하면 삭제/붙여넣기 완료 후 오른쪽 목록이 빈 배경으로 깜박일 수 있다.
- Shell 충돌 해결 UI, 진행률 UI, 권한 상승 UI에서 사용자가 취소하거나 Shell이 일부 작업 중단을 보고하면 사용자 오류 대화상자를 띄우지 않고 파일 작업 취소로 정리한다.
- 드롭/붙여넣기 copy의 실행 취소 후보는 모든 예상 대상이 작업 전에는 없었고 작업 후에는 존재하는 경우에만 만든다. 대상 존재 여부가 불명확하거나, 기존 대상이 있었거나, 일부 예상 대상만 생성된 경우에는 실행 취소 후보를 만들지 않는다.
- 드롭/붙여넣기 move의 실행 취소 후보와 완료 선택 항목은 모든 예상 대상이 생성되고 모든 원본 경로가 작업 후 사라진 경우에만 만든다. 원본이 남아 있거나 일부 대상만 이동된 경우에는 부분 성공으로 간주해 실행 취소 후보와 완료 선택을 만들지 않는다.
- 드래그 중 폴더 트리 항목 위에서 hover 대상이 일정 시간 유지되면 해당 항목을 자동 확장한다. hover 대상이 바뀌면 pending 기준 시각을 새 대상 기준으로 갱신한다.
- 드래그 중 폴더 트리 또는 파일 목록의 위/아래 가장자리에서 hover하면 해당 컨트롤을 한 줄씩 자동 스크롤한다. 스크롤은 대상 탐색 보조 동작이며 드롭 대상 의미는 계속 `NavigationLocation`으로 판정한다.
- 드래그 종료, 취소, 드롭 거부, 드롭 대상 이탈, drop target 해제 시 hover pending 상태와 자동 스크롤/확장 timer를 정리한다.

현재 구현 범위:

- 1차 구현은 앱이 드롭 대상이 되는 흐름과 앱 내부 파일 목록에서 폴더 대상으로 끌어 놓는 흐름을 지원한다.
- 3차 구현은 앱 밖으로 파일을 끌어 내보내는 Shell drag source를 지원한다.
- 드래그 중 Windows 커서 효과는 현재 대상과 수정 키에 따라 복사, 이동, 불가로 갱신한다.
- 4차 안정화는 내부 드롭, 외부 Shell 드롭 수신, 앱 밖 Shell drag source를 같은 파일 작업 경험으로 통합한다. 자동화 가능한 copy/move 효과, 이동 금지, 선택 스냅샷, 완료 후 갱신 대상, worker 중복 실행 방지는 단위 테스트로 고정한다.
- 5차 릴리즈 후보 안정화는 한글/유니코드, verbatim 긴 경로 prefix, UNC 공유, 다중 선택 대량 드래그의 `HGLOBAL` 크기 방어, Shell 오류 메시지 매핑, worker 실행 중 중복 드래그/드롭 거부, 앱 종료 중 완료 메시지 처리를 검증 대상으로 둔다.
- 6차 최종 릴리즈 마감은 새 기능 추가 없이 blocker 결함만 수정하며, OLE/COM 수명, Shell 실패 복구, worker 중복 보호, 종료 중 완료 메시지, 릴리즈 빌드 산출물을 잠금 검증 대상으로 둔다.
- 폴더 트리 자동 확장은 현재 구현 범위에 포함하며, 같은 트리 항목 위에서 일정 시간 hover하면 해당 항목을 확장한다.
- 8차 UX 보강은 폴더 트리 hover 자동 확장을 timer 기반으로 보강하고, TreeView/ListView 위/아래 가장자리 자동 스크롤을 추가한다. 자동화 가능한 hover pending과 스크롤 방향 결정은 순수 로직 테스트로 고정한다.
- 9차 대량 선택 안정화는 `CF_HDROP` 전체 메모리 상한, 드래그 시작 전 preflight, 중복 소스 제거, 대량/긴 경로 경계값 테스트를 잠금 검증 대상으로 둔다.
- 10차 손실 방지 점검은 Shell 취소의 오류 대화상자 억제, copy 부분 성공 시 실행 취소 후보 미생성, move 원본 잔존 시 실행 취소 후보와 완료 선택 미생성, 공유 위반 메시지 매핑, worker 실행 중 추가 드롭/드래그 거부를 잠금 검증 대상으로 둔다.
- 11차 코드 경계 정리는 새 기능 없이 `CF_HDROP` 생성/읽기 공통화, 외부 drop 기본 효과 결정의 domain 고정, 파일 작업 worker 전송 요청 타입 정리, 관련 회귀 테스트 재배치를 검증 대상으로 둔다.
- 12차 릴리즈 전 최종 리뷰는 새 기능 없이 파일 손실, 크래시, OLE/COM 수명, worker 경합, 문서 불일치 같은 blocker만 수정하고 자동 검증과 수동 Shell 상호운용 체크리스트를 릴리즈 PR 기준으로 정리한다.
- 세밀한 삽입 위치 표시는 현재 범위에 포함하지 않는다. j3Files의 현재 드롭 대상은 폴더 위치이며, 파일 목록 행 사이 삽입 의미가 아직 도메인 규칙으로 정의되어 있지 않기 때문이다.
- 검색 결과에서 파일 작업을 직접 수행하는 정책, Shell namespace/PIDL 기반 고급 드래그 소스, 파일 작업 worker 다중 큐, Shell 진행률 UI의 앱 내부 상태 통합은 후속 범위로 둔다.

### 5.5 파일 실행

사용자는 파일을 기본 앱 또는 연결된 프로그램으로 실행할 수 있다.

규칙:

- 파일 실행은 `ShellExecuteEx`를 사용한다.
- 기본 앱으로 열기는 Windows Shell의 기본 연결 프로그램을 따른다.
- 연결 프로그램 선택은 Windows Open With UI를 사용하며, `ShellExecuteEx`의 `openas` verb로 호출한다.
- 목록 항목 더블클릭과 메뉴의 열기 명령은 `entry`에서 직접 Shell API를 호출하지 않고 `app` 유스케이스로 전달한다.
- Win32 파일 목록의 마우스 더블클릭과 키보드 실행은 `LVN_ITEMACTIVATE`를 단일 항목 활성화 신호로 사용하며, `NM_DBLCLK`를 별도 실행 신호로 중복 처리하지 않는다.
- 실행 실패 시 내부 오류와 사용자 메시지를 분리한다.

### 5.6 파일 속성 보기

사용자는 파일 또는 폴더의 Windows 속성 창을 열 수 있다.

규칙:

- 속성 보기는 `ShellExecuteEx`를 사용한다.
- 속성 보기는 `properties` verb를 사용한다.
- 애플리케이션이 자체 속성 창을 우선 제공하지 않는다.
- 메뉴의 속성 명령은 `entry`에서 직접 Shell API를 호출하지 않고 `app` 유스케이스로 전달한다.
- Shell 호출 실패 시 사용자에게 속성 창을 열 수 없다는 메시지를 표시한다.

### 5.7 목록 정렬

사용자는 목록을 이름, 크기, 갱신일, 종류로 정렬할 수 있다.

규칙:

- 정렬 기준은 탭 또는 전역 사용자 설정으로 유지할 수 있다.
- 정렬 방향은 오름차순과 내림차순을 지원할 수 있다.
- 파일 목록의 Name, Type, Size, Updated 열 헤더를 클릭하면 해당 정렬 기준을 적용한다.
- 현재 정렬 기준과 같은 열 헤더를 다시 클릭하면 오름차순과 내림차순을 전환한다.
- 종류 정렬은 Windows Shell 또는 확장자 기반 종류 정보를 기준으로 한다. 초기 구현은 확장자 기반 종류 이름을 사용한다.
- 정렬은 안정 정렬이어야 하며, 선택한 정렬 기준 값이 같은 항목의 기존 순서를 임의로 바꾸지 않는다.
- 크기 정렬에서 폴더 크기는 즉시 계산하지 않는 것을 기본으로 한다.

### 5.8 탭 관리

사용자는 여러 탭을 열어 서로 다른 경로를 동시에 탐색할 수 있다.

기능:

- 새 탭 열기
- 탭 닫기
- 탭 전환
- 탭 순서 변경
- 폴더를 새 탭으로 열기
- 닫은 탭 다시 열기
- 탭바 우클릭 메뉴 표시
- 앱 재실행 시 이전 탭 복원 옵션

규칙:

- 각 탭은 현재 경로를 독립적으로 유지한다.
- 각 탭은 뒤로/앞으로 탐색 기록을 독립적으로 유지한다.
- 각 탭은 정렬 기준, 검색 상태, 선택 항목을 독립적으로 유지한다.
- 새 탭 열기와 폴더를 새 탭으로 열기는 요청 위치 접근 가능성을 확인한 뒤 새 탭을 활성화한다.
- entry UI의 새 탭 명령은 현재 활성 탭의 현재 경로에서 새 탭을 시작한다.
- 탐색 툴바의 `+` 버튼은 entry UI의 새 탭 명령과 같은 동작으로 현재 활성 탭의 현재 경로에서 새 탭을 시작한다.
- 탭 순서 변경은 열린 탭 목록 안에서만 허용하며, 활성 탭 식별자는 유지한다.
- 탭 라벨은 현재 경로의 표시 이름만 사용하고 순번 접두사는 붙이지 않는다.
- 현재 선택된 탭은 탭 라벨을 굵게 표시해 다른 열린 탭과 쉽게 구분할 수 있어야 한다.
- 탭을 닫을 때 해당 탭의 상태는 닫은 탭 복원 목록에 보관할 수 있다.
- 닫은 탭 다시 열기는 최근에 닫은 탭을 우선 복원한다.
- 닫은 탭 복원은 복원 대상의 현재 경로 접근 가능성을 확인한 뒤 열린 탭 목록 끝에 추가하고 활성화한다.
- 탭바 우클릭 메뉴는 상단 `Tabs` 메뉴와 같은 명령 집합을 현재 마우스 위치에 표시한다.
- 특정 탭 위에서 탭바 우클릭 메뉴를 열면 해당 탭을 먼저 활성화한 뒤 탭 명령을 적용한다.
- 앱 재실행 시 이전 탭 복원은 사용자 옵션으로 제공한다.
- 이전 탭 복원 시 접근할 수 없는 경로는 건너뛰거나 오류 상태의 탭으로 표시한다.

### 5.9 경로 탐색

사용자는 주소 표시줄과 탐색 명령으로 현재 위치를 변경할 수 있다.

기능:

- 주소 표시줄 직접 입력
- 뒤로 이동(`arrow_back_ios` 아이콘)
- 앞으로 이동(`arrow_forward_ios` 아이콘)
- 위로 이동(`arrow_upward` 아이콘)
- 새로 고침(`refresh` 아이콘)
- 새 탭 열기(`+` 버튼)
- 주소 표시줄 이동 실행(`arrow_right_alt` 아이콘)
- 빠른 접근 또는 북마크
- 드라이브 목록
- 바탕화면, 다운로드, 문서 접근
- `\\server\share` 형식 네트워크 경로 접근

규칙:

- Win32 탐색 툴바의 뒤로, 앞으로, 위로, 새로 고침, 이동 실행 버튼은 Google Fonts Material Symbols Outlined PNG 아이콘을 리소스로 포함해 사용하고, 아이콘 색상은 `#3a86ff`로 유지한다.
- 주소 표시줄은 로컬 경로와 UNC 경로를 모두 허용한다.
- 뒤로/앞으로 이동은 현재 탭의 탐색 기록만 사용한다.
- 위로 이동은 현재 위치의 부모 위치로 이동한다.
- 드라이브와 Windows 기본 폴더는 Windows API 또는 Shell API를 통해 조회한다.
- 빠른 접근 항목은 존재 여부와 접근 가능 여부를 표시 또는 갱신할 수 있어야 한다.
- 북마크 항목으로 이동하는 동작은 현재 탭에서 해당 경로로 이동하는 것과 동일하게 처리한다.

### 5.9.1 키보드 단축키

사용자는 Windows 파일 탐색기에서 기대하는 주요 키보드 단축키로 탐색, 검색, 파일 작업, 탭 관리를 수행할 수 있다.

탐색 단축키:

- `Alt+Left`는 현재 탭의 뒤로 이동을 실행한다.
- `Alt+Right`는 현재 탭의 앞으로 이동을 실행한다.
- `Alt+Up`은 현재 탭의 상위 폴더 이동을 실행한다.
- `F5`는 현재 폴더를 새로 고친다.
- `Ctrl+L`과 `Alt+D`는 주소 표시줄에 포커스를 두고 전체 경로를 선택한다.
- `Ctrl+F`와 `F3`은 검색창을 표시하고 검색어 입력칸에 포커스를 둔다.
- 검색어 입력칸에 포커스가 있을 때 `Esc`는 검색창을 닫는다.

파일 목록 단축키:

- `Ctrl+C`는 선택한 파일 항목을 Windows 파일 클립보드에 복사 작업으로 기록한다.
- `Ctrl+X`는 선택한 파일 항목을 Windows 파일 클립보드에 이동 작업으로 기록한다.
- `Ctrl+V`는 Windows 파일 클립보드의 파일 항목을 현재 탭 위치에 복사 또는 이동한다.
- `Delete`는 선택한 파일 항목을 휴지통으로 삭제한다.
- `Shift+Delete`는 선택한 파일 항목을 휴지통을 거치지 않고 삭제한다.
- `F2`는 선택한 단일 파일 항목의 이름 변경 편집을 시작한다.
- `Ctrl+A`는 현재 파일 목록의 모든 항목을 선택한다.
- `Ctrl+Click` 다중 선택은 Win32 ListView의 기본 다중 선택 동작을 사용한다.
- `Ctrl+Z`는 앱이 안전하게 추적할 수 있는 최근 이름 변경 또는 붙여넣기 작업을 되돌린다. 휴지통 삭제와 영구 삭제처럼 복구 대상이 외부 상태에 있는 작업은 초기 구현에서 실행 취소 기록으로 만들지 않는다.

탭 단축키:

- `Ctrl+T`는 현재 위치에서 새 탭을 연다.
- `Ctrl+W`는 현재 탭을 닫는다.
- `Ctrl+Tab`은 다음 탭으로 이동한다.

규칙:

- 주소 표시줄과 검색창 같은 편집 컨트롤에 포커스가 있을 때는 `Ctrl+C`, `Ctrl+X`, `Ctrl+V`, `Ctrl+A`, `Ctrl+Z`, `Delete`, `Shift+Delete`, `F2`가 텍스트 편집 기본 동작을 유지해야 한다.
- 단축키 입력은 메뉴와 버튼 명령과 같은 `app` 유스케이스를 호출해야 하며, entry가 파일 시스템 변경 작업을 직접 수행하지 않는다.
- 파일 클립보드는 Windows Shell이 이해하는 `CF_HDROP`와 `Preferred DropEffect` 형식을 사용해 다른 파일 관리자와 복사/잘라내기 상태를 교환할 수 있어야 한다.

### 5.9.2 왼쪽 폴더 트리

사용자는 왼쪽 패널의 폴더 트리에서 주요 위치와 확장한 폴더의 하위 폴더를 선택해 현재 탭의 탐색 위치를 변경할 수 있다.

기능:

- Home, Desktop, Downloads, Documents 바로가기
- 드라이브 루트 목록
- 사용자가 저장한 북마크 목록
- 확장한 폴더의 하위 폴더

규칙:

- 폴더 트리는 폴더, 드라이브, Windows 기본 폴더, 네트워크 공유처럼 탐색 가능한 위치만 표시한다.
- 폴더 트리는 파일 항목을 표시하지 않는다.
- 확장한 폴더의 자식 항목은 파일 목록 조회 결과 중 폴더 항목만 사용해 만든다.
- 폴더 트리 항목은 내부 식별자로 `NavigationLocation`을 보관하고, 선택 시 이 값을 그대로 `app` 탐색 유스케이스에 전달한다.
- 폴더 트리 항목의 표시 이름은 사용자에게 보여 주기 위한 값일 뿐이며, 표시 문자열을 경로로 역변환해 탐색 대상을 만들지 않는다.
- Home, Desktop, Downloads, Documents는 `KnownFolderKind`와 `NavigationLocation::KnownFolder`를 사용해 Windows가 해석한 실제 경로와 연결한다.
- 드라이브 항목은 드라이브 루트 탐색 위치만 사용한다.
- 북마크 항목은 북마크의 표시 이름과 대상 `NavigationLocation`을 그대로 사용하며, 북마크가 접근 불가능해도 트리 항목 자체는 보존할 수 있다.
- 트리 확장 가능 여부는 `FolderTreeItem.has_children()` 기반 UI 힌트이며, 실제 접근 가능 여부는 선택 또는 확장 시 파일 시스템 검증 결과로 판정한다.
- Win32 왼쪽 폴더 트리는 `app`의 `folder_tree_roots()` 결과로 루트 항목을 구성하고, 각 TreeView 항목의 내부 값은 표시 문자열이 아니라 `NavigationLocation` 매핑 식별자이다.
- TreeView 항목 확장 시점에만 `folder_tree_children()`으로 하위 폴더를 조회해 붙이며, 이미 로드한 항목은 중복으로 다시 붙이지 않는다. 확장 결과가 비어 있으면 해당 TreeView 항목의 하위 항목 힌트를 제거한다.
- TreeView 항목 선택은 현재 탭에 대한 일반 탐색과 동일하게 처리하므로, 검색 결과 표시 중에도 선택 탐색은 활성 검색 상태를 취소하고 일반 폴더 목록으로 전환한다.
- TreeView 항목 우클릭은 해당 항목의 `NavigationLocation`을 대상으로 오른쪽 파일 목록과 같은 Windows Shell 컨텍스트 메뉴를 표시하며, 우클릭 자체만으로 현재 탭의 탐색 위치를 변경하지 않는다.
- TreeView 항목 선택 또는 확장 중 접근 권한 부족, 네트워크 실패, 없는 경로 오류가 발생하면 `ExplorerError.user_message()`로 사용자 메시지를 표시하고 현재 탭의 현재 경로는 유지한다.
- TreeView 선택 동기화는 저장된 `NavigationLocation`의 실제 경로가 현재 경로와 같도록 루트부터 필요한 부모 항목을 지연 로드하고 확장한 뒤 선택한다.
- TreeView 선택 동기화의 경로 비교는 Windows 경로의 대소문자, 구분자, 끝 구분자 차이를 정규화해 비교한다.
- 오른쪽 파일 목록, 주소 표시줄, 메뉴, 탭 전환처럼 현재 탐색 위치가 바뀌는 흐름은 왼쪽 폴더 트리를 현재 위치까지 확장하고, 대상 폴더 자체도 확장 가능한 경우 펼친다.
- TreeView 동기화 중 현재 경로에 대응하는 항목을 찾을 수 없거나 자동 하위 폴더 조회가 실패하면 선택 없음 상태로 두며, 일반 탐색 성공 자체를 실패로 취급하지 않는다.
- 북마크 추가처럼 루트 구성이 바뀌는 동작은 TreeView 루트를 다시 구성한다.
- 새 폴더 생성, 삭제, 붙여넣기, 이름 변경처럼 표시 중인 폴더 구조가 바뀔 수 있는 파일 작업은 전체 TreeView를 다시 만들지 않고 영향 폴더와 일치하는 로드된 항목의 자식만 갱신한 뒤 현재 위치 선택을 동기화한다.
- 탭 전환과 일반 화면 갱신은 현재 탐색 위치에 맞춰 TreeView 선택만 동기화한다.
- 사용자는 왼쪽 폴더 트리와 오른쪽 파일 목록 사이의 분할선을 마우스로 끌어 좌우 영역 폭을 조정할 수 있다.

### 5.10 북마크 관리

사용자는 자주 방문하는 폴더, 드라이브, Windows 기본 폴더, 네트워크 경로를 북마크로 저장하고 다시 이동할 수 있다.

기능:

- 현재 경로를 북마크로 추가
- 선택한 폴더를 북마크로 추가
- 북마크 이름 변경
- 북마크 삭제
- 현재 위치와 같은 대상의 북마크 삭제
- 왼쪽 폴더 트리의 북마크 항목에서 북마크 삭제
- 북마크 순서 변경
- 북마크 선택 시 해당 경로로 이동

규칙:

- 북마크 대상은 탐색 위치로 해석 가능한 경로여야 한다.
- 북마크는 사용자가 명시적으로 추가한 항목으로 빠른 접근과 구분한다.
- 북마크의 표시 이름은 사용자가 지정할 수 있으며, 지정하지 않으면 대상 경로의 마지막 구성 요소를 기본 이름으로 사용한다.
- 같은 대상 경로를 중복 추가할지 여부는 정책으로 결정하되, 기본 구현은 중복 추가를 막고 기존 북마크를 선택한다.
- 새 북마크가 추가되면 사용자 설정 저장을 예약해 앱 재실행 후에도 유지한다.
- 북마크 순서는 사용자가 변경할 수 있으며 앱 재실행 후에도 유지한다.
- 북마크 삭제는 대상 파일 또는 폴더를 삭제하지 않고 북마크 항목만 제거한다.
- 폴더 트리의 북마크 항목 컨텍스트 메뉴는 대상 폴더의 Shell 삭제 명령과 혼동되지 않도록 앱 전용 북마크 메뉴를 표시한다.
- 북마크 대상 경로가 존재하지 않거나 접근할 수 없어도 북마크 항목 자체는 보존한다.
- 접근할 수 없는 북마크를 선택하면 경로 접근 실패 메시지를 표시하고 현재 탭의 기존 경로는 유지한다.
- 네트워크 경로 북마크는 연결 지연이나 실패가 있을 수 있으므로 일반 로컬 경로 실패와 구분 가능한 내부 오류를 남긴다.

### 5.11 Windows 컨텍스트 메뉴

사용자는 파일 또는 폴더에서 Windows 컨텍스트 메뉴를 열 수 있다.

규칙:

- 컨텍스트 메뉴 항목은 Windows Shell이 제공하는 메뉴를 우선 사용한다.
- 선택 항목이 여러 개인 경우 Windows Shell 다중 선택 컨텍스트 메뉴 동작을 따른다.
- 오른쪽 파일 목록의 빈 영역에서 우클릭하면 선택 항목이 없어도 현재 탭 탐색 위치의 Windows Shell 폴더 배경 컨텍스트 메뉴를 표시한다.
- 메뉴 명령 실행 후 목록 갱신이 필요한 경우 현재 폴더를 새로 고친다.

### 5.12 표시 옵션

사용자는 숨김 파일과 시스템 파일 표시 여부를 설정할 수 있다.

규칙:

- 숨김 파일 표시 옵션은 숨김 속성 파일 표시 여부를 제어한다.
- 시스템 파일 표시 옵션은 시스템 속성 파일 표시 여부를 제어한다.
- 시스템 파일 표시 옵션은 사용자가 의도적으로 켜는 설정으로 취급한다.
- 옵션 변경 시 현재 목록을 다시 필터링하거나 새로 고친다.

### 5.12.1 외형 테마

사용자는 `View > Theme` 메뉴에서 앱 외형 테마를 선택할 수 있다.

규칙:

- 기본 테마는 `Light`이다.
- 선택 가능한 테마는 `Light`, `Classic Dark`, `Sepia Teal`, `Graphite`, `Forest`, `Steel Blue`이다.
- `Classic Dark`, `Sepia Teal`, `Graphite`, `Forest`, `Steel Blue`는 j3TreeText의 같은 이름 팔레트를 기준으로 한다.
- 테마 선택은 사용자 설정으로 저장하고 앱 재실행 후 복원한다.
- 저장값이 올바르지 않으면 앱 시작 실패로 처리하지 않고 `Light`로 fallback한다.
- 테마 변경은 탭, 탐색 기록, 선택 항목, 검색 상태, 파일 작업 상태를 변경하지 않는다.
- Win32가 직접 그리는 컨트롤은 가능한 범위에서 제목 표시줄, 창 배경, 폴더 트리, 탭, 입력창, 버튼, 파일 목록 색상을 함께 갱신한다.

### 5.12.2 외형 글꼴

사용자는 `View > Font...` 메뉴에서 시스템에 설치된 글꼴과 글꼴 크기를 선택할 수 있다.

규칙:

- 기본 글꼴은 Windows 기본 GUI 글꼴이다.
- 글꼴 선택은 Windows 표준 Font dialog를 사용하며, 설치된 화면 글꼴만 선택 대상으로 한다.
- 글꼴 크기는 6pt부터 36pt까지 허용한다.
- 선택한 글꼴 이름과 크기는 사용자 설정으로 저장하고 앱 재실행 후 복원한다.
- 저장된 글꼴 이름이 없거나 글꼴 크기 값이 올바르지 않으면 앱 시작 실패로 처리하지 않고 Windows 기본 GUI 글꼴로 fallback한다.
- 글꼴 변경은 탭, 탐색 기록, 선택 항목, 검색 상태, 파일 작업 상태를 변경하지 않는다.
- Win32 글꼴 핸들 생성과 수명 관리는 `platform::win32_ui`에 격리하고, `entry`는 도메인 설정값만 전달한다.

### 5.13 검색

사용자는 현재 폴더에서 파일을 검색할 수 있다.

기능:

- 현재 폴더 내 검색
- `sub` 하위 폴더 포함 검색 옵션
- 파일명 검색
- 검색 중 취소

규칙:

- 기본 검색 범위는 현재 폴더이다.
- 하위 폴더 포함 옵션이 켜져 있으면 현재 폴더 아래를 재귀적으로 검색한다.
- entry UI는 `find` 실행 버튼, `sub` 체크박스, `Search > Include Subfolders` 메뉴 상태를 조합해 검색 범위를 결정한다.
- `sub` 체크박스와 `Search > Include Subfolders` 메뉴 체크 표시는 같은 검색 범위 상태를 보여야 하며, 어느 쪽을 클릭해도 다음 메뉴 표시 전에 동기화되어야 한다.
- 검색 입력줄은 기본 탐색 중에는 숨기고, 사용자가 검색을 시작하려고 열었거나 검색 실행/결과/취소 상태가 현재 탭에 남아 있을 때만 표시한다.
- `Search > Find` 명령은 검색 입력줄이 숨겨져 있으면 먼저 입력줄을 표시하고 검색어 입력에 포커스를 둔다.
- 검색 결과 또는 취소 상태에서 취소 명령은 검색 상태를 지우고 검색 입력줄을 다시 숨긴다.
- 검색어 입력칸에 포커스가 있을 때 `Esc`는 검색 상태를 지우고 검색 입력줄을 숨긴다. 실행 중인 검색은 취소 신호를 보낸 뒤 일반 목록으로 돌아간다.
- 검색 취소 버튼은 Google Fonts Material Symbols Outlined `cancel` 아이콘을 리소스로 포함해 사용하고, 아이콘 색상은 `#3a86ff`로 유지한다.
- 검색은 파일명 기반으로 수행한다.
- `SearchCriteria`는 파일명 검색어와 검색 범위를 포함한다.
- 검색어 판정은 `domain`의 순수 함수로 수행하고, Windows 파일 시스템 순회와 분리한다.
- 검색 작업은 취소 가능해야 한다.
- entry UI는 탭별로 실행 중인 검색 worker를 하나만 유지하고, 기존 worker가 종료되기 전 새 검색 요청은 대기 검색으로 보관한다.
- 검색 worker의 `JoinHandle`은 entry가 보관하고, 완료 메시지 처리 또는 완료된 worker 정리 시 회수한다.
- Win32 디렉터리 열거 호출은 동기 API이므로 호출 중인 `FindFirstFileW`/`FindNextFileW`는 즉시 중단하지 못하며, 취소는 호출 반환 후 관찰한다.
- 검색 중 접근 권한이 없는 폴더는 전체 검색을 실패시키지 않고 건너뛰며 내부 오류로 기록할 수 있다.
- 검색 결과는 현재 탭의 경로 상태와 구분되는 검색 결과 상태로 표시한다.
- 검색 진행 상태는 방문한 폴더 수, 확인한 항목 수, 일치 항목 수, 건너뛴 폴더 수를 포함한다.
- 검색 완료, 검색 취소, 부분 결과는 실패와 구분한다.

### 5.14 상단 메뉴 정보 구조

상단 메뉴는 데스크톱 애플리케이션 관례와 사용자 의도에 맞춰 명령을 묶는다.

규칙:

- `File`은 현재 선택 항목 또는 현재 폴더에 대한 파일 시스템 작업과 앱 종료를 포함한다.
- `Edit`은 실행 취소, 클립보드, 전체 선택처럼 편집 및 선택 명령을 포함한다.
- `View`는 새로 고침, 정렬, 숨김/시스템 파일 표시, 외형 설정처럼 현재 표시 방식을 바꾸는 명령을 포함한다.
- `Go`는 뒤로, 앞으로, 상위 폴더, Windows 기본 폴더, 드라이브처럼 탐색 위치 이동 명령을 포함한다.
- entry UI에서 `Go > Back` 또는 `Go > Forward`가 기록이 없는 상태로 실행되면 사용자 동작은 no-op으로 처리하고 오류 로그를 남기지 않는다. 도메인 계층의 기록 pop 오류는 내부 불일치 진단 의미로 유지한다.
- `Bookmarks`는 사용자가 명시적으로 저장한 탐색 위치와 북마크 추가/삭제 명령만 포함한다.
- `Tabs`는 현재 탭 조작과 탭 시작 동작 설정을 포함하며, 앱 재실행 동작은 `Startup` 하위 메뉴로 분리한다.
- `Search`는 검색 입력 표시, 하위 폴더 포함, 검색 취소, 검색 닫기처럼 검색 상태 관련 명령을 포함한다.
- 휴지통 삭제 메뉴 라벨은 영구 삭제와 혼동되지 않도록 `Move to Recycle Bin`으로 표시한다.
- 기본 폴더와 드라이브는 Windows Quick Access와 동기화하지 않고 `Go` 메뉴의 빠른 이동 항목으로 제공한다.

## 6. Windows 연동 규칙

### 6.1 Shell 파일 작업

복사, 이동, 삭제, 이름 변경은 애플리케이션이 직접 파일 바이트 복사나 재귀 삭제를 구현하지 않고 Windows Shell 시스템 기능을 사용한다.

기본 API:

- `IFileOperation`

적용 작업:

- 복사
- 이동
- 삭제
- 이름 변경

Windows Shell에 위임할 동작:

- 파일명 충돌 처리
- 권한 상승
- 진행률 표시
- 사용자 취소
- 휴지통 이동
- 네트워크 위치와 특수 폴더 처리

### 6.2 Shell 실행과 속성

파일 실행과 속성 보기는 `ShellExecuteEx`를 사용한다.

적용 작업:

- 기본 앱으로 열기
- 연결 프로그램으로 열기
- 속성 보기

연결 프로그램 선택은 Windows Open With UI를 사용한다.

규칙:

- 앱 창이 준비된 뒤 실행되는 `ShellExecuteEx` 호출은 owner window를 전달해 Windows Shell이 표시하는 연결 프로그램 선택, 속성, 권한 관련 UI가 앱 창에 종속되도록 한다.
- `ShellExecuteEx` 호출 전 COM STA 초기화를 수행하고, 실패 시 내부 오류에는 `CoInitializeEx` 또는 `ShellExecuteExW` API 이름과 Windows 오류 코드 또는 HRESULT를 보존한다.
- 기본 앱으로 파일을 열 때는 파일의 부모 폴더를 작업 디렉터리로 전달해 실행 파일과 스크립트의 상대 경로 동작이 Windows 파일 탐색기 실행 흐름과 어긋나지 않게 한다.

### 6.3 파일 아이콘

파일과 폴더 아이콘은 Windows Shell에서 조회한다.

규칙:

- 확장자, 폴더, 드라이브, 네트워크 위치, 특수 폴더의 아이콘은 Windows Shell 결과를 따른다.
- 아이콘 조회 실패 시 기본 파일 또는 폴더 아이콘으로 대체한다.
- 아이콘 캐시는 UI 성능을 위해 사용할 수 있으나, Shell 결과와 불일치할 수 있음을 고려해 무효화 경로를 둔다.

### 6.4 관리자 권한

권한 상승이 필요한 작업은 Windows Shell 또는 OS 권한 상승 흐름을 사용한다.

규칙:

- 애플리케이션은 권한 부족을 `panic`으로 처리하지 않는다.
- 가능한 경우 Shell 작업이 제공하는 권한 상승 UI를 따른다.
- 권한 상승이 거부되면 사용자가 취소했거나 권한이 부족하다는 메시지를 표시한다.

### 6.5 경로와 문자 인코딩

긴 경로와 한글/유니코드 파일명을 지원한다.

규칙:

- Windows 경로는 유니코드 경로로 처리한다.
- 내부 경로 표현은 손실 변환을 피한다.
- 사용자 표시용 문자열과 내부 경로 값을 분리한다.
- `\\server\share` 형식 UNC 경로를 유효한 탐색 위치로 취급한다.

### 6.6 애플리케이션 아이콘

앱 식별 아이콘은 저장소 루트의 `icon.ico`를 단일 원본으로 사용한다.

규칙:

- 실행 파일 아이콘은 Windows 리소스로 빌드 산출물에 포함한다.
- 프로그램 창 아이콘은 같은 리소스를 Win32 창 클래스의 큰 아이콘과 작은 아이콘으로 로드한다.
- 아이콘 리소스 ID는 entry와 리소스 스크립트가 같은 값을 사용해야 한다.

## 8. 에러 처리 정책

복구 가능한 오류는 `Result`로 전달한다. 내부 원인과 사용자 메시지는 분리한다.

내부 오류에 포함할 정보:

- 작업 종류
- 대상 경로와 대상 폴더. 복사/이동처럼 대상 폴더가 별도로 있는 작업은 원본 경로와 대상 폴더를 함께 보존한다.
- Windows API 이름
- Windows 오류 코드 또는 HRESULT. HRESULT 기반 API는 원시 HRESULT를 보존하고 Win32 코드로 해석 가능한 경우 파생 코드도 함께 보존한다.
- 사용자 취소 여부
- 권한 상승 필요 여부

사용자 메시지 원칙:

- 내부 API 이름이나 원시 오류 코드를 그대로 노출하지 않는다.
- 사용자가 다음 행동을 판단할 수 있는 문장으로 표시한다.
- 예: "파일을 삭제할 수 없습니다. 권한이 없거나 파일이 사용 중일 수 있습니다."

취소 처리:

- 사용자가 취소한 작업은 실패와 구분한다.
- 취소된 검색이나 파일 작업은 가능한 한 부분 결과와 부분 완료 상태를 명확히 표시한다.
- entry UI는 사용자 취소로 분류된 오류를 일반 실패 메시지 대화상자로 표시하지 않는다.
- 권한 상승이 필요하거나 거부된 경우 내부 오류에는 권한 상승 필요 여부를 남기고, 사용자에게는 관리자 권한이 필요하다는 메시지만 표시한다.

## 10. 비기능 요구사항

- 파일 목록 표시와 아이콘 조회는 UI를 장시간 블로킹하지 않아야 한다.
- 검색과 긴 파일 작업은 취소 가능해야 한다.
- 파일 작업은 Windows Shell의 진행률과 충돌 해결 동작을 우선 사용한다.
- 접근 권한 부족, 파일 사용 중, 네트워크 끊김 같은 오류를 복구 가능한 오류로 처리한다.
- 경로와 파일명은 한글과 유니코드를 손실 없이 처리한다.
- 런타임 경로에서 `unwrap()` 또는 `expect()`로 파일 시스템 오류를 처리하지 않는다.
- Windows 전용 구현은 공용 도메인 규칙과 분리한다.
- 드래그앤드롭 파일 작업은 UI 스레드를 장시간 블로킹하지 않고 기존 파일 작업 worker와 Shell 진행률 UI를 사용해야 한다.
- 프로세스 DPI awareness는 Win32 창 생성 전에 설정하며, 실패하더라도 내부 진단을 남기고 Windows 기본 스케일링으로 계속 실행한다.
- 기본 창 크기, 최소 창 크기, 탐색 버튼, 여백, 검색 행, 왼쪽 폴더 트리 폭, 파일 목록 열 너비처럼 실제 픽셀 값인 UI 옵션은 96 DPI 기준 값에서 현재 DPI 기준으로 보정한다.
- 목록 중심 탐색 도구의 창 이동 안정성을 우선하므로 모니터 이동 중 창 geometry를 반복 재설정하지 않는다.
- 릴리즈 빌드 보조 스크립트는 Windows와 Linux에서 같은 명령으로 실행할 수 있어야 하며, 빌드 성공 후 산출물 폴더를 운영체제의 기본 파일 관리자에서 열어야 한다.
- Windows 릴리즈 GUI 실행은 별도 콘솔 창을 생성하지 않아야 한다.

| 릴리즈 빌드 성공 후 `build_release.py`는 Cargo 릴리즈 산출물 폴더를 Windows Explorer 또는 Linux 기본 파일 관리자에서 연다. | 사용자가 빌드 완료 후 실행 파일 위치를 바로 확인할 수 있어야 하기 때문이다. |
| Windows 릴리즈 빌드의 `src/main.rs`는 `windows_subsystem = "windows"`를 사용하고 디버그 빌드는 기본 콘솔 subsystem을 유지한다. | Explorer에서 릴리즈 실행 파일을 열 때 검은 콘솔 창을 만들지 않으면서, 개발 중에는 표준 오류 로그를 콘솔에서 확인할 수 있어야 하기 때문이다. |
| 복사, 이동, 삭제, 이름 변경은 `IFileOperation`을 기본으로 사용한다. | Windows의 충돌 처리, 권한 상승, 진행률, 취소, 휴지통 동작을 재사용하기 위해서이다. |
| `IFileOperation`은 가능한 경우 앱의 owner window를 `SetOwnerWindow`로 전달한다. | Shell 진행률, 충돌 해결, 권한 상승, 취소 UI가 앱 창에 종속되어 표시되어야 하기 때문이다. |
| `IFileOperation`의 Copy Engine 전용 `HRESULT`는 원시 값을 보존하되 취소, 권한 상승, 권한 부족, 이름 충돌처럼 사용자 행동에 필요한 분류는 Win32 코드로 함께 매핑한다. | Shell 파일 작업 실패를 진단 가능하게 남기면서 사용자 메시지는 취소와 실패, 권한과 충돌을 구분해야 하기 때문이다. |
| 파일 실행과 속성 보기는 `ShellExecuteEx`를 사용한다. | Windows 파일 연결과 속성 창 동작을 표준 Shell 동작과 일치시키기 위해서이다. |
| `ShellExecuteEx`는 가능한 경우 앱의 owner window를 전달하고 COM STA 초기화 후 호출한다. | Shell 확장, Open With UI, 속성 창, 권한 관련 UI가 앱 창과 올바르게 연결되고 Shell verb 구현이 COM 기반인 경우에도 안정적으로 동작해야 하기 때문이다. |
| 연결 프로그램 선택은 `ShellExecuteEx`의 `openas` verb를 사용한다. | Windows Open With UI를 직접 구현하지 않고 Shell이 제공하는 선택 흐름을 사용하기 위해서이다. |
| 파일/폴더 Windows 컨텍스트 메뉴는 `IShellFolder::GetUIObjectOf(IContextMenu)`와 `TrackPopupMenu`를 통해 표시한다. | Shell 확장과 다중 선택 컨텍스트 메뉴 동작을 Windows Shell에 위임하기 위해서이다. |
| 파일 목록 빈 영역의 Windows 컨텍스트 메뉴는 현재 폴더의 `IShellFolder::CreateViewObject(IContextMenu)`와 `TrackPopupMenu`를 통해 표시한다. | 항목 선택이 없는 폴더 배경 메뉴도 Windows Explorer와 같은 Shell 메뉴 항목과 Shell 확장 동작을 사용해야 하기 때문이다. |
| 컨텍스트 메뉴 명령이 실행되면 현재 폴더를 새로 고침 대상으로 취급한다. | Shell 명령이 파일 생성, 삭제, 이름 변경, 속성 변경을 수행했는지 앱이 일반적으로 판정하기 어렵기 때문이다. |
| 드래그앤드롭은 파일 작업 유스케이스의 별도 입력 방식으로 모델링하고, 실제 복사/이동은 기존 `IFileOperation` 경로를 재사용한다. | 충돌 처리, 권한 상승, 진행률, 취소, 오류 매핑을 붙여넣기와 동일하게 유지하고 entry가 파일 시스템 변경을 직접 수행하지 않게 하기 위해서이다. |

| 설정 저장은 같은 디렉터리의 임시 파일에 완전히 쓴 뒤 Windows 원자적 교체로 대상 설정 파일을 바꾼다. | 앱 종료나 쓰기 중단 중 설정 파일이 부분 기록 상태로 남는 일을 줄이기 위해서이다. |
| 설정 파일을 읽거나 파싱하지 못하면 사용자에게 경고하고 안전한 기본값으로 시작한다. | 손상된 설정 파일 때문에 앱 시작 자체가 실패하지 않아야 하며, 내부 진단은 보존하되 사용자는 탐색을 계속할 수 있어야 하기 때문이다. |
| 저장되는 사용자 설정은 열린 탭 목록, 활성 탭, 닫은 탭 복원 목록, 북마크 목록, 숨김 파일 표시, 시스템 파일 표시, 외형 테마, 외형 글꼴, 시작 폴더 설정, 이전 탭 복원 옵션이다. | 현재 전역 상태로 모델링된 사용자 선택과 세션을 앱 재실행 후에도 유지하기 위해서이다. |
| 외형 테마 저장 키는 `appearance_theme`이며 저장값은 `light`, `classic-dark`, `sepia-teal`, `graphite`, `forest`, `steel-blue` 중 하나이다. `dark_theme` 및 j3TreeText 호환 키 `appearance.theme`, `appearance.dark_theme`도 읽을 수 있다. | j3Files 설정 형식을 유지하면서 j3TreeText의 테마 팔레트와 수동 설정 이전을 쉽게 하기 위해서이다. |
| Win32 테마 적용은 `platform::win32_ui`에서 제목 표시줄 dark mode, 컨트롤 선호 테마, TreeView/ListView 색상 메시지, `WM_ERASEBKGND`와 `WM_CTLCOLOR*` 브러시 처리를 묶어 수행한다. | raw Win32/GDI 핸들과 브러시 수명 책임을 entry와 domain에서 분리하기 위해서이다. |
| 외형 글꼴 저장 키는 `appearance_font_family`, `appearance_font_size`이다. 글꼴 이름은 UTF-16 코드 유닛 hex로 저장하고 크기는 포인트 정수로 저장한다. | 새 직렬화 의존성을 추가하지 않으면서 한글/유니코드 글꼴 이름을 손실 없이 왕복시키기 위해서이다. |
| Win32 글꼴 선택은 표준 `ChooseFontW` dialog를 사용하고, 적용은 `WM_SETFONT`와 소유 `HFONT` 수명 관리를 `platform::win32_ui`에 둔다. | 설치된 시스템 글꼴 조회 UI와 GDI 핸들 책임을 직접 구현하지 않고 Windows 경계에 격리하기 위해서이다. |
| 기본 UI 글꼴은 Windows에서 `Malgun Gothic` 9pt를 DPI 기준 `HFONT`로 생성해 적용한다. | 한글 UI와 파일명 표시를 선명하게 유지하고, DPI awareness 이후에도 기본 컨트롤 글자가 과도하게 작아지지 않게 하기 위해서이다. |
| 프로세스 DPI awareness는 `SYSTEM_AWARE`를 먼저 시도하고, 실패 시 `PER_MONITOR_AWARE`, `PER_MONITOR_AWARE_V2`, `SetProcessDpiAwareness(SYSTEM_DPI_AWARE)`, `SetProcessDpiAwareness(PER_MONITOR_DPI_AWARE)`, `SetProcessDPIAware()` 순서로 fallback한다. | j3Files는 TreeView, ListView, 버튼 중심의 관리 도구라서 모니터 간 이동 중 보이는 목록 양과 창 크기의 안정성이 모니터별 동적 재스케일보다 우선이기 때문이다. |
| DPI 관련 Win32 함수 중 OS 버전에 따라 없을 수 있는 함수는 `GetProcAddress`로 동적 조회한다. | 오래된 Windows에서 프로세스 로드 단계부터 실패하지 않고 가능한 API부터 순서대로 시도하기 위해서이다. |
| `DpiMetrics`와 `UiScale`은 `platform::win32_ui`에 두고, entry는 픽셀 상수를 직접 곱하지 않고 값 객체를 통해 변환한다. | DPI 계산식과 Win32 DPI 조회 책임을 UI 플랫폼 경계에 모아 레이아웃 코드가 DPI 정책 세부사항에 묶이지 않게 하기 위해서이다. |
| `WM_DPICHANGED`를 받더라도 제안된 창 크기 rect로 geometry를 되먹이지 않고, DPI가 실제로 바뀐 경우에만 폰트, 아이콘, 픽셀 기반 레이아웃 옵션을 다시 적용한다. | per-monitor fallback으로 실행되더라도 모니터 경계에서 창 크기 흔들림과 반복 레이아웃 재계산을 줄이기 위해서이다. |
| 네이티브 크기/이동 루프 중 `WM_DPICHANGED`를 받으면 DPI 메트릭과 레이아웃 갱신을 즉시 실행하지 않고 pending으로 표시한 뒤 `WM_EXITSIZEMOVE`에서 실제 현재 창 DPI와 client rect 기준으로 한 번만 갱신한다. | mixed-DPI 모니터 경계에서 Windows의 size/move modal loop와 앱의 동기 레이아웃 재계산이 같은 UI thread를 막아 창 이동이 끊기거나 좌표가 튀는 일을 줄이기 위해서이다. |
| 네이티브 크기/이동 루프 중 DPI 갱신이 pending이면 `WM_SIZE`에서 child control 배치를 즉시 다시 계산하지 않는다. | 이전 DPI 메트릭 기준의 중간 크기 메시지로 TreeView, Tab, ListView를 반복 이동시키지 않고 종료 시점의 실제 client rect에 맞춰 안정적으로 재배치하기 위해서이다. |
| 저장 탭 상태는 현재 경로, 뒤로/앞으로 기록, 정렬 기준, 정렬 방향을 포함하고 검색 상태와 선택 항목은 저장하지 않는다. | 검색 실행과 선택 항목은 일시적 UI 상태이며, 재실행 후에는 탐색 가능한 위치와 기본 목록 표시를 안정적으로 복원하는 것이 우선이기 때문이다. |
| 이전 탭 복원 옵션이 켜진 시작 복원에서는 현재 경로에 접근 가능한 열린 탭만 복원하고 접근할 수 없는 열린 탭은 건너뛴다. | 초기 구현에는 오류 상태 탭 UI가 없으므로 시작 실패를 피하면서 접근 가능한 세션을 최대한 복원하기 위해서이다. |
| 숨김 파일과 시스템 파일 표시 옵션은 초기 구현에서 전역 사용자 설정으로 저장한다. | 탭별·폴더별 정책보다 단순하며 전체 탐색 경험에서 일관된 기대를 제공하기 때문이다. |
| 북마크 중복 판단과 폴더 트리 선택 동기화는 `NavigationLocation`의 Windows 경로 정규화 비교를 사용한다. | 같은 탐색 위치가 표기 차이만으로 중복 저장되거나 현재 위치와 트리 선택이 어긋나는 일을 막기 위해서이다. |
| 폴더 크기는 목록 표시와 크기 정렬 시 즉시 계산하지 않는다. | 큰 폴더와 네트워크 위치에서 목록 표시가 오래 블로킹되는 일을 피하기 위해서이다. |
| Windows 기본 폴더와 드라이브 빠른 이동은 `Go` 메뉴의 앱 자체 항목으로 유지하고 Windows Quick Access와 동기화하지 않는다. | Windows Shell의 자동 추천/고정 위치와 앱의 명시적 북마크 의미가 다르므로, 동기화는 별도 요구사항으로 분리해야 하기 때문이다. |
| 검색은 취소 가능한 작업으로 모델링한다. | 하위 폴더 검색과 네트워크 경로 검색이 오래 걸릴 수 있기 때문이다. |
| 검색 실행은 앱 상태 전이와 백그라운드 파일 시스템 작업으로 분리한다. | Win32 UI 스레드가 긴 검색 동안 블로킹되지 않고, 완료 메시지가 현재 탭 검색 상태에 반영되어야 하기 때문이다. |
| 현재 검색 작업은 `std::thread` worker, `Arc<AtomicBool>` 취소 플래그, window message 기반 진행/완료 통지로 실행한다. | 기존 Win32 message loop와 별도 async 런타임 없이 취소 가능한 장시간 파일 시스템 순회를 UI 스레드 밖으로 분리하기 위해서이다. |
| entry는 검색 worker의 `JoinHandle`을 보관하고 탭별 실행 worker를 하나로 제한한다. | 느린 네트워크 또는 장치 경로에서 취소된 worker가 Win32 동기 열거 호출 안에 머무를 수 있으므로, 재검색이 새 스레드를 계속 만들지 않고 완료 시 handle을 명시적으로 회수해야 하기 때문이다. |
| Win32 command id 분기는 명령 라우터가 사용자 의도로 변환하고, `MainWindow`는 변환된 의도를 기존 앱 유스케이스와 UI 갱신에 연결한다. | 새 메뉴나 단축키를 추가할 때 command id 해석과 실제 상태 변경 책임을 분리해 회귀 범위를 줄이기 위해서이다. |
| 검색, 목록 조회, 파일 작업 worker의 실행 상태와 pending 완료 메시지는 백그라운드 작업 컨트롤러가 소유한다. | UI entry가 worker 수명, 메시지 rollback, generation 토큰 증가 규칙을 직접 섞어 갖지 않게 하기 위해서이다. |
| 검색용 디렉터리 순회는 Win32 열거 결과를 한 항목씩 처리하며 항목 처리 전후로 취소 플래그를 확인한다. | 큰 폴더 하나를 전부 읽는 동안 취소와 진행률 통지가 지연되지 않게 하고, 취소된 검색을 실패와 구분해 부분 결과로 마무리하기 위해서이다. |
| 검색 결과 상태는 일반 폴더 목록 상태와 별도로 탭의 `SearchState`에 저장한다. | 검색 결과를 표시해도 현재 경로와 탐색 기록이 변경되지 않아야 하기 때문이다. |
| 하위 폴더 검색 중 읽을 수 없는 폴더는 건너뛰고 내부 진단으로 기록한다. | 권한 부족 또는 네트워크 일부 오류 때문에 전체 검색을 실패시키지 않기 위해서이다. |
| 현재 일반 폴더 목록 갱신과 아이콘 조회는 UI 스레드에서 동기 실행한다. | 초기 구현의 상태 모델을 단순하게 유지하기 위한 선택이지만, 큰 폴더나 느린 네트워크 위치에서는 백그라운드 목록 작업으로 분리해야 할 수 있다. |
| 내부 오류와 사용자 메시지를 분리한다. | 진단 가능성과 사용자 이해 가능성을 동시에 확보하기 위해서이다. |
| 초기 GUI 엔트리포인트는 Win32 창, 메뉴, 툴바, 주소 입력, 왼쪽 폴더 트리, 자세히 보기 파일 목록을 제공한다. | 앱 상태 전이와 Win32 호출 래퍼를 섞지 않고 탐색 흐름을 고정하기 위해서이다. |
| 시작 시 폴더 선택 대화상자를 자동 표시하지 않고, 사용자가 저장한 시작 폴더 옵션을 우선 후보로 사용한다. | 앱 실행 흐름을 방해하지 않으면서 사용자가 원하는 시작 위치를 재실행 후에도 유지하기 위해서이다. |
| 시작 위치는 저장된 시작 폴더, 사용자 프로필 폴더, 현재 디렉터리 순서의 후보를 사용한다. | 사용자가 명시한 작업 위치를 우선하되, 접근할 수 없는 경우에도 접근 가능한 위치에서 탭 상태를 시작하기 위해서이다. |
| 드라이브 목록은 `GetLogicalDriveStringsW`, Windows 기본 폴더는 `SHGetKnownFolderPath`로 조회한다. | 드라이브 문자와 사용자 프로필 기본 폴더 리디렉션을 Windows가 제공하는 실제 위치와 일치시키기 위해서이다. |
| 초기 빠른 이동 UI는 메뉴로 제공하고, 선택한 항목은 `app` 계층의 현재 탭 탐색 유스케이스로 이동한다. | Win32 GUI의 기존 구조를 크게 흔들지 않으면서 주소 입력, 목록 진입, 빠른 이동이 동일한 접근성 검증과 오류 메시지를 공유하도록 하기 위해서이다. |
| 새 폴더 생성은 이름 검증 후 `CreateDirectoryW`를 호출하고, Shell 충돌 해결 UI를 띄우지 않는다. | 단일 폴더 생성은 앱 상태 갱신과 오류 매핑이 핵심이며, 이름 충돌은 자동 해결하지 않고 `Result` 오류로 전달해야 하기 때문이다. |

## 12. 후속 백로그와 미정 사항

- 폴더별 보기 옵션 저장 여부. 현재 구현은 정렬 기준과 방향을 탭 상태로, 숨김 파일 표시와 시스템 파일 표시 옵션을 전역 사용자 설정으로 유지한다.
- 일반 폴더 목록 조회와 아이콘 조회를 검색처럼 백그라운드 작업으로 전환할지 여부. 현재 구현은 UI 스레드 동기 갱신이며 큰 폴더, 느린 드라이브, 네트워크 위치에서 블로킹 위험이 남아 있다.
- 검색 결과에서 파일 작업을 수행할 때 원본 위치 목록 갱신, 선택 유지, 여러 원본 부모 폴더 새로 고침 정책
- 즉시 삭제 기능을 제공할지 여부
- Shell interop 수동 테스트 자동화 가능 범위. Explorer/Desktop 양방향 OLE 드래그, UAC prompt, Shell 충돌 해결 UI, 장시간 Shell 진행률 UI는 비대화형 테스트에서 자동화하기 어렵기 때문에 수동 smoke와 제한적 통합 테스트로 분리한다.
- 드래그 중 세밀한 삽입 위치 표시 정책. 현재 구현은 폴더 위치 드롭과 edge 자동 스크롤만 제공하며, 파일 목록 행 사이 삽입 의미는 정의하지 않는다.
- ListView/TreeView 자동 스크롤 속도와 edge 폭의 실제 장치별 튜닝. 현재 구현은 DPI 기준 edge 폭과 100ms line-scroll timer를 사용한다.
- Shell namespace/PIDL 기반 고급 드래그 소스 지원 여부. 현재 드롭 수신과 앱 밖 드래그 소스는 파일 시스템 경로를 담은 `CF_HDROP`를 기준으로 한다.
- 파일 작업 worker와 Shell 진행률 UI 개선. 현재 파일 작업 worker는 한 번에 하나만 실행하고 `IFileOperation` 진행률/충돌/권한 상승 UI를 Shell에 맡기며, 앱 내부에는 진행 중/종료 대기 상태를 별도 상태 줄로 표시한다.

## 13. 초기 구현 범위

- Rust 애플리케이션 크레이트로 구성한다.
- `src/main.rs`는 플랫폼 분기 엔트리포인트만 담당하고, `src/windows_main.rs`는 Win32 GUI 엔트리포인트로서 창 생성, 메시지 루프, 메뉴, 툴바, 주소 입력, 왼쪽 폴더 트리, 파일 목록 영역, 사용자 메시지 표시만 담당한다.
- `src/app`은 탭 탐색, 목록 조회, 북마크, Shell 작업 유스케이스를 조합한다.
- `src/domain`은 `NavigationLocation`, `FileItem`, `TabState`, `SortKey`, `SortDirection`, `SearchCriteria`, `BookmarkItem`, `ExplorerError`를 포함한 핵심 타입과 상태 전이 규칙을 담당한다.
- `src/domain`은 `FolderTreeItem`을 통해 왼쪽 폴더 트리 항목의 표시 이름과 내부 탐색 위치 식별자를 분리한다.
- `src/infra`는 파일 시스템 목록 조회, 시작 위치 후보 조회, Shell 게이트웨이 구현을 담당한다.
- `src/platform`은 `windows-sys` 기반 Win32/Shell/UI 호출 래퍼만 담당한다.
- `src/platform/win32_ui`는 raw Win32 handle을 불투명 handle 래퍼로 감싸 entry에 노출하고, `WNDPROC` callback 진입점에서만 raw handle을 래퍼로 변환한다.
- `src/platform/win32_ui`는 DPI awareness 설정, `GetDpiForWindow`/시스템 DPI fallback 조회, `DpiMetrics`, `UiScale`, DPI 기준 폰트 생성을 함께 제공한다.
- `src/infra/shell.rs`는 Shell owner window를 포인터 크기 값으로만 보관하고, raw `HWND` 변환과 Shell API 호출은 `src/platform`에 둔다.
- 초기 Shell 연동은 `ShellExecuteExW` 기반 파일 열기와 속성 보기, `IFileOperation` 기반 복사/이동/휴지통 삭제/이름 변경, `GetLogicalDriveStringsW`, `SHGetKnownFolderPath`, `GetFileAttributesW`를 제공한다.
- 초기 파일 시스템 목록 조회는 `windows-sys`의 `FindFirstFileW`/`FindNextFileW`와 `WIN32_FIND_DATAW`를 사용한다.
- `src/platform`은 Win32 파일 속성, 파일 크기, `FILETIME`, UTF-16 파일명 변환을 격리한다.
- `src/platform`은 새 폴더 생성을 위한 `CreateDirectoryW` 호출과 Windows 오류 코드 보존을 담당한다.
- `src/platform/hdrop.rs`는 클립보드와 OLE drag data가 공유하는 `CF_HDROP` 생성/읽기, `HGLOBAL` 소유권, 항목 수/경로 길이/전체 byte size 검증을 담당한다.
- 실행 파일 아이콘은 `app.rc`가 `icon.ico`를 Windows 리소스로 포함하고, 초기 GUI 창 클래스는 같은 리소스를 창 아이콘으로 사용한다.
- `src/infra`는 Win32 조회 결과를 `FileItem`으로 변환하고 숨김/시스템 속성 필터링과 도메인 정렬 규칙을 적용한다.
- 초기 GUI 파일 목록은 Win32 ListView의 자세히 보기 모드로 이름, 크기, 갱신일, 종류 열을 표시한다.
- 주소 입력 이동, 위로 이동, 새로 고침은 `app` 계층의 탐색 유스케이스를 통해 현재 탭 상태를 변경한다.
- 복사, 이동, 휴지통 삭제, 이름 변경은 상위 계층에 `Result` 경계를 노출하고, `platform`의 `IFileOperation` 래퍼를 통해 실행한다.
- 드래그앤드롭 초기 구현은 `platform`의 OLE 드롭 대상 래퍼가 파일 목록과 폴더 트리 드롭을 감지하고, `entry`가 이를 기존 파일 작업 worker와 `app` 복사/이동 유스케이스로 전달하는 구조로 둔다.
- 드래그앤드롭 2차 안정화는 `domain`의 순수 드롭 효과 판단과 이동 금지 규칙을 테스트하고, `platform`의 OLE 드롭 대상 래퍼가 드래그 중 대상별 copy/move/none 효과와 폴더 트리 hover 확장을 처리하는 구조로 둔다.
- 드래그앤드롭 3차 구현은 `platform`의 OLE drag source 데이터 객체가 내부 드래그 식별자와 Shell `CF_HDROP`를 제공하고, `entry`가 ListView 선택 스냅샷과 완료 결과별 갱신만 연결하는 구조로 둔다.
- 드래그앤드롭 4차 안정화는 외부 `Preferred DropEffect` 반영, Shell 호출 직전 move 금지 재검증, 작업 영향 폴더 기반 갱신, file operation worker 중복 실행 거부를 테스트로 고정하는 구조로 둔다.
- 드래그앤드롭 5차 릴리즈 후보 안정화는 Shell drag source가 목적지 의존 기본 효과를 강제하지 않도록 `Preferred DropEffect`를 내보내지 않고, `CF_HDROP` 생성 전 항목 수와 UTF-16 경로 길이 및 `HGLOBAL` 크기 overflow를 검증하는 구조로 둔다.
- 드래그앤드롭 6차 최종 릴리즈 마감은 `DoDragDrop` 전 대량 선택 검증, OLE drop 이벤트 전달 실패 시 불가 효과 반환, 한국어 파일 작업 상태 메시지, 문서화된 수동 검증 체크리스트와 릴리즈 빌드 흐름을 확인하는 구조로 둔다.
- 드래그앤드롭 8차 UX 보강은 `domain`의 hover pending/자동 스크롤 방향 결정과 `platform`/entry-adjacent timer, TreeView/ListView hit-test, line-scroll 호출을 분리하는 구조로 둔다.
- 드래그앤드롭 11차 경계 정리는 `domain`의 외부 drop 기본 효과 순수 규칙, `platform::hdrop`의 `CF_HDROP` 공통 처리, entry-adjacent `FileOperationRequest::Transfer` worker 요청을 기준 구조로 둔다.
- 검색은 entry가 worker thread를 시작하고, `app`이 만든 `SearchRequest`와 `infra`의 파일 시스템 게이트웨이를 사용해 실행한 뒤, 진행/완료 window message로 UI 상태에 반영한다.
